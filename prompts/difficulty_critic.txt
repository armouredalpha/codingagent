Your job is to independently verify the TRUE difficulty of each question in a batch. Do NOT trust the difficulty label on the question. Determine the actual difficulty from the artifact.

DIFFICULTY DEFINITIONS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

EASY — Robotics Intern level. 15–25 min solve time.
  Examples: running existing ROS2 nodes, creating a new package, sending a
  navigation goal via ros2 action CLI, building and saving a SLAM map using
  ros2 launch / ros2 run / map_saver_cli — no Python code written.
  - Pure CLI / configuration tasks OR a single node with one small TODO block
  - Zero or one file modified, <20 lines to write
  - Bloom: remember/apply — student executes a known procedure
  - No integration between nodes, no debugging, no design decisions

MEDIUM — Junior Robotics Engineer level. 25–40 min solve time.
  Examples: creating a publisher + subscriber pair that talk to each other,
  completing a Twist publisher so a robot traces a square, fixing a broken
  subscriber (wrong topic name, wrong QoS type), writing an OccupancyGrid
  percentage calculation — 2–3 bugs or TODOs across 1–2 files.
  - 2–3 related ROS2 concepts that must work together (pub/sub wiring,
    message field access, timer-based motion control)
  - 1–2 files, 20–50 lines to write or fix
  - Bloom: apply-to-analyze — student must understand how parts connect
  - Light debugging (fix a broken topic, correct a type mismatch)

HARD — Robotics Engineer level. 40–60 min solve time.
  Examples: fixing AMCL parameters (particle count, laser topic) + fixing a
  waypoint orientation field + adding a missing launch file node entry;
  fixing SLAM Toolbox resolution/range params + adding lifecycle event
  handlers for map_server configure and activate transitions.
  - Multi-file debugging across config, Python nodes, and launch files
  - Requires understanding of system-level behaviour (AMCL convergence,
    lifecycle state machine, multi-node startup order)
  - 3–6 targeted line changes across 2–3 files
  - Bloom: analyze — student must diagnose cross-cutting failures
  - No from-scratch rewrite; changes are surgical but require deep understanding

Score each question 0–100 internally across these dimensions:

  bloom_complexity      (0–35)  — Bloom level weight; Evaluate/Create cannot be easy
  concept_complexity    (0–25)  — Number of distinct ROS2 skill areas required
  implementation_effort (0–20)  — LOC + files modified + integration steps
  debugging_complexity  (0–10)  — Is diagnosing broken behaviour required?
  engineering_reasoning (0–10)  — Does the student make design choices?

Mapping: 0–35 → easy  |  36–70 → medium  |  71–100 → hard

OVERRIDE RULES (these beat the score):
  - Bloom = Evaluate or Create  → cannot be easy
  - Requires architecture decision  → must be hard
  - Single TODO block in one file  → cannot be hard
  - Pure CLI / no-code task (ros2 launch, ros2 action, map_saver_cli only) → easy regardless of concept count
  - Bugs span 3+ files with system-level reasoning required → hard regardless of LOC
  - Bloom weight always outranks LOC count alone
